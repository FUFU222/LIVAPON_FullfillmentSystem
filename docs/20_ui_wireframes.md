# UI Wireframes Spec (2026-05-06)

shadcn/ui + Tailwind をベースにした白×黒＋アクセント `#801010` のモダン UI を想定。フォントはシステムフォント、余白は `gap-4~6` を基準に揃える。

---

## 0. ランディング `/`
- ヒーロー: H1「セラー向け配送コンソール」、サブコピー、CTA ボタン 2 つ（黒背景サインイン / アウトライン申請）。
- オンボーディングフロー: 「申請 → 承認 → 利用開始」の 3 カード。カード間は水平グラデーションライン。モバイル時は縦積み + 縦ライン。
- トップバナー: `Official Partner Access` のピル型バッジ。背景は白→薄灰グラデーション。

## 1. 認証・申請フロー
### `/sign-in`
- 中央揃えカード (max-w-md)。
- フォーム: メール・パスワード + サインインボタン。成功時は役割に応じてリダイレクト。

### `/(public)/apply`
- ヘッダー: 「セラー利用申請」。
- フォーム: 会社名、担当者名(任意)、メール、パスワード/確認、備考、規約チェック。
- 送信中モーダル: 全画面半透明 + メッセージ。
- 完了モーダル: 成功文 + 「審査状況を見る」CTA（`/pending`）。

### `/pending`
- カード (max-w-2xl) で審査待ちメッセージと連絡先メールリンク。

## 2. セラーコンソール
### `/orders`
- カードレイアウト。ヘッダー: タイトル + `OrdersRefreshButton`（トースト付き）。desktop は上部ナビ、mobile は下部ナビから履歴へ移動。
- フィルター: 検索テキスト（注文番号/顧客名）、ステータスセレクト。
- desktop: テーブル列は 注文番号 / 顧客名 / 商品数 / 追跡番号（複数は Badge 群） / ステータス / 最終更新。
- mobile: テーブルを横スクロールさせず、注文カードで注文番号・顧客名・ステータス・対象ライン・主要操作を縦に整理する。
- テーブル下に発送登録パネル（ラインアイテム選択リスト + 追跡番号 + 配送業者セレクト + 確認モーダル）。
- 送信成功時は「発送情報を登録しました」Toast、入力リセット、`router.refresh()`。ジョブ進捗や polling UI は表示しない。

### `/orders/shipments`
- ヘッダー: 戻るリンク、タイトル、更新ボタン。
- desktop: テーブルは 出荷番号、追跡番号、配送会社、同期ステータス（Badge）、発送日時、対象ライン、管理者操作を横並びにする。
- mobile: 発送単位のカード表示。長い Shopify エラーは全文を縦に流し、ボタン・入力欄が画面幅を超えないようにする。
- 同期失敗時の再同期・手動対応済み化・Fulfillment ID 紐付けは admin 向け導線。セラーセルフサービスの再送/取消は置かない。

### `/import`
- legacy / reference route。ヘッダーとモバイルナビには出さない。
- 再公開する場合は、サンプル CSV ダウンロード → ファイル選択 → プレビュー → 実行の責務と、即時受付型配送登録との関係を再設計する。

### `/vendor/profile`
- 見出しカード: Avatar + 会社名 + セラーコード。
- フォーム: 会社名、担当者、メール、任意パスワード。フォーム送信で Toast を表示。

## 3. 管理者コンソール
### `/admin`
- 3 カラムカード（審査申請 / 最近の注文 / 新規セラー）。
- 各カードは件数 Badge + リスト/Alert。Link ボタンで詳細ページへ。

### `/admin/applications`
- 上段カード: pending セラー申請を `VendorApplicationCard` で縦並び。
- 下段カード: 最近審査した申請をテーブル表示（申請ID / 会社名 / セラーコード / 担当者 / メール / ステータス / 審査日時）。

### `/admin/vendors`
- ヘッダー: タイトル。
- 本体: `VendorBulkDeleteForm` のテーブル。
  - 列: 選択チェック / セラー名 / コード / 申請ステータス Badge / Auth 連携状態 / メール / 登録日 / アクション。
  - 行クリックでモーダル。モーダルに `AdminVendorDetail`（プロフィール、サマリー、申請履歴）。
  - 一括削除ボタンはテーブル上部右配置、実行時は確認モーダル。

### `/admin/orders`
- カード + テーブル。列は注文番号 / セラー名 / ステータス / 顧客名 / 更新日時。

## 4. UI パターン指針
- Buttons: 主要＝黒背景、アウトライン＝灰枠、危険操作は赤枠/テキスト。
- Alerts: shadcn Alert (`variant="success|destructive|default"`)。一覧空の場合は成功バナーを表示。
- Toast: 右上固定、`title` のみ 1 行。長時間処理は `duration: Infinity` + 手動 dismiss。
- テーブル: `thead` 薄灰背景、`tbody` ホバーで `bg-slate-50`。
- モーダル: `@/components/ui/modal` を共通利用。Esc/Backdrop で閉じ、フォーカストラップ必須。
- ナビゲーション: グローバル blocking spinner は使わない。route loading は静的 skeleton を表示し、既に表示可能な UI を塞がない。
- アイコン: mobile nav や短い操作ボタンは lucide-react の既存アイコンを優先し、長いテキストラベルで画面幅を圧迫しない。

## 5. タイポグラフィ & 配色
| 用途 | メモ |
| ---- | ---- |
| H1 | `text-3xl sm:text-5xl`, `font-semibold`, tracking-tight |
| セクションタイトル | `text-2xl font-semibold` |
| テーブル本文 | `text-sm text-slate-600` |
| 補足文 | `text-xs text-slate-500` |
| アクセントカラー | `#801010` をバッジ・ピル・強調ラインに限定 |

## 6. レスポンシブ基準
- 最小幅 320px。主要業務画面はカード化を優先し、横スクロール前提にしない。例外的に表形式が必要な管理テーブルだけ横スクロールを許容する。
- CTA ボタンは全幅 (`w-full`) をデフォルトにし、`sm:` から横並び。
- 重要情報は折り畳まない。必要に応じ `data-[state=open]` で詳細を Accordion 表示する。
- モバイル下部ナビがある画面では `env(safe-area-inset-bottom)` を考慮し、フォーム送信ボタンやフッターが隠れない余白を確保する。

## 7. QA チェックリスト
- [ ] サインイン〜承認〜利用開始の導線が 3 クリック以内で理解できる。
- [ ] セラーの主要操作（発送登録・発送履歴確認・修正申請・プロフィール変更）が 2 ステップ以内。
- [ ] Toast/Alert がアクセシブル（aria-live, aria-role 適切）。
- [ ] `OrdersRefreshButton` 実行後に Loading 表現と Toast が出る。
- [ ] 管理者ページで権限不足時は `/orders` へリダイレクトされる。
- [ ] 320px / 390px / 768px / 1024px / 1440px でヘッダー・下部ナビ・テーブル/カード・モーダルが画面幅をはみ出さない。
